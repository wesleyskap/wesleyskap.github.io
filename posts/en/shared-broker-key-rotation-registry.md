## Security and Contract Governance in Microservices

In robust distributed systems, data governance involves two main aspects:
1. **Contract Governance:** How do we guarantee that the structure of an event payload generated by Team A is perfectly understood and processed by Team B's consumers dynamically and centrally?
2. **Granular Data Security:** How do we ensure that highly sensitive data (PII, payment information) is encrypted with specific keys, while general messages use a shared key? And how do we rotate those keys without causing consumer crashes when processing historical messages remaining in the queues?

The **SharedBroker** gem expands its validation and security borders with support for a remote **HTTP Schema Registry** (using standard JSON Schema specs) and a **Topic-Specific Key Rotation Strategy**.

---

## 1. HTTP Schema Registry: Centrally Validating Distributed Contracts

While local validation using `dry-schema` is perfect for fast checks during local development, complex enterprise ecosystems require a central contract registry (Schema Registry).

`SharedBroker` supports an HTTP provider for JSON Schema. During initialization or on-demand, the gem fetches schemas from a centralized registry API and validates them against outbound and inbound payloads. To prevent network latency bottlenecks during runtime validations, the gem caches schemas in memory with a configurable TTL.

### Configuring the HTTP Provider

```ruby
# config/initializers/shared_broker.rb

SharedBroker::SchemaRegistry.provider = SharedBroker::SchemaRegistry::Providers::Http.new(
  url: "https://schema-registry.corp.internal",
  headers: { "Authorization" => "Bearer governance-secret-token" },
  cache_ttl: 300 # Cache schemas in-memory for 5 minutes
)
```

Whenever the application publishes or consumes an event (such as `user.created`), `SharedBroker` validates the payload by fetching the corresponding JSON Schema definition from `https://schema-registry.corp.internal/schemas/user.created.json`.

## 2. Granular Encryption and Key Rotation

Traditional queue encryption using a single global key is fragile: if the secret leaks, all data across all queues is compromised. Furthermore, changing the global key (Key Rotation) is a technical headache, as old messages encrypted with the previous key that haven't been consumed yet will fail to decrypt.

To solve this, we implemented a **Key Provider Registry** in `SharedBroker`. It handles rotation and granularity by defining:
* Mapping active keys to topics using wildcard patterns (e.g., `payment.*` uses a finance key, other topics use a general key).
* A dictionary of historical keys (`keys`) identified by a unique ID (`_key_id`).
* Automatically embedding the `_key_id` tag in the message envelope during publication.

### Key Provider Registry Logic

```ruby
module SharedBroker
  module KeyProvider
    class Registry
      def initialize(keys:, active_keys:)
        @keys = keys
        @active_keys = active_keys
      end

      # Identify which key is active when publishing to a topic
      def active_key_for(topic)
        match = @active_keys.keys.find { |pattern| File.fnmatch?(pattern, topic) }
        key_id = @active_keys[match] || "*"
        
        [key_id, @keys[key_id]]
      end

      # Retrieve the correct historical key for decryption based on the envelope ID
      def find_key(key_id)
        @keys[key_id] || raise("Cryptographic key not found in registry: #{key_id}")
      end
    end
  end
end
```

### Configuring the Key Registry

```ruby
# Initialize registry mapping active keys to topic patterns
key_registry = SharedBroker::KeyProvider::Registry.new(
  keys: {
    "v1"            => "historical_global_32_byte_key_a",
    "v2"            => "current_global_32_byte_key_b_b",
    "finance_key_1" => "secure_finance_32_byte_key_here"
  },
  active_keys: {
    # Financial topics use the exclusive finance key
    "payment.*" => "finance_key_1",
    # Any other topic uses the latest version of the global key (v2)
    "*"         => "v2"
  }
)

# Inject the key manager into the client
SPOT_BROKER = SharedBroker::Client.new(
  adapter: BROKER_ADAPTER,
  key_provider: key_registry
)
```

With this workflow, when the producer publishes `payment.processed`, the payload is encrypted using `finance_key_1` and the metadata tag `_key_id: "finance_key_1"` is attached. If there are older messages in the queue tagged with `_key_id: "v1"`, the consumer dynamically fetches the correct historical key from the registry and decrypts the message seamlessly.

## Conclusion

Securing the boundaries of our message queues doesn't have to be a heavy or complex process to maintain. By utilizing central validations via Schema Registry and granularity with key rotation in `SharedBroker`, we elevate the security and integrity of our event-driven architectures without cluttering our business logic.

### Technical Terms Demystified
- **JSON Schema:** A declarative language based on JSON for validating the structure, types, and constraints of structured data.
- **TTL (Time to Live):** A duration limit after which cached data is considered expired and discarded.
- **Glob Pattern:** Simple wildcard string matching using characters like `*` (matches zero or more characters) and `?` for quick filtering.
